# 网络联机五子棋对战系统 - 开发文档

## 项目概述

### 1.1 项目简介
本项目是一款基于 Qt 5.15+ 开发的网络联机五子棋对战系统，支持单机AI对战、在线玩家对战、观战系统等功能。采用 C/S 架构，使用 Reactor 网络模型、Redis 缓存、MySQL 数据库和 Protobuf 序列化协议。

### 1.2 核心功能
- **单机模式**：与AI进行五子棋对战
- **联机对战**：通过房间号或随机匹配与其他玩家对战
- **社交系统**：大厅聊天、私聊、房间内聊天
- **房间系统**：支持房间创建、加密房间、房间号加入
- **排行榜**：显示玩家胜利场次、胜率等数据
- **观战系统**：允许其他玩家观看正在进行的对局

### 1.3 技术栈

| 技术 | 用途 |
|------|------|
| Qt | GUI框架、网络通信 |
| C++ | 编程语言 |
| Protobuf | 数据序列化 |
| MySQL | 用户数据、战绩存储 |
| Redis | 在线用户、房间缓存 |
| Reactor模式 | 网络I/O多路复用 |

### 2.3 服务器架构

#### 2.3.1 Reactor 网络模型

服务器采用 **单 Reactor + 线程池** 模型：

```mermaid
graph LR
    subgraph MainThread["主线程 (Reactor)"]
        EventLoop[EventLoop<br/>事件循环]
        Poller[Poller<br/>epoll/select]
        Channels[多个 Channel<br/>fd监听]
        Callback[事件回调<br/>处理器]
    end

    subgraph WorkerThreads["工作线程池"]
        ThreadPool[Thread Pool<br/>线程池]
        Business[业务处理<br/>Protobuf解析<br/>数据库操作]
        Result[结果返回<br/>EventLoop]
    end

    EventLoop --> Poller
    Poller --> Channels
    Channels -->|事件就绪| Callback
    Callback -->|投递任务| ThreadPool
    ThreadPool --> Business
    Business --> Result
    Result -.->|异步回调| EventLoop

    style MainThread fill:#ffebee
    style WorkerThreads fill:#e8f5e9
```

**核心组件**：

| 组件 | 职责 |
|------|------|
| EventLoop | 事件循环，驱动 Poller |
| Poller | 封装 epoll/select，监听文件描述符 |
| Channel | 封装 fd 和事件回调函数 |
| Acceptor | 监听新连接，创建 Connection |
| Connection | 管理单个 TCP 连接，处理读写 |
| ThreadPool | 处理 CPU 密集型任务 |

## 第三章 数据流程设计

### 3.1 用户登录流程

**流程说明**：用户通过客户端输入用户名和密码，服务器验证后返回登录结果，成功则将用户信息缓存到Redis并返回Token。

```mermaid
sequenceDiagram
    participant C as 客户端<br/>(Client)
    participant S as 服务器<br/>(Server)
    participant R as Redis缓存<br/>(在线状态)
    participant M as MySQL数据库<br/>(用户数据)

    Note over C: 用户输入账号密码
    C->>S: 1. 发送登录请求<br/>LoginRequest (username, password)

    Note over S: 查询用户信息，通过加密算法验证密码
    S->>M: 2. 查询用户记录<br/>SELECT * FROM users WHERE username=? and password=?
    M-->>S: 3. 返回用户数据<br/>User Record (id, ...)


    alt 登录成功
        Note over S,R: 将用户状态写入Redis缓存
        S->>R: 5  存储用户连接信息<br/>SET online:user:{id} {conn_info}
        S->>R: 6  添加到在线用户集合<br/>ZADD online_users {timestamp} {user_id}

        Note over S: 生成会话Token
        S-->>C: 4  返回登录成功响应<br/>LoginResponse (success, user_info, token)

        Note over C: 保存Token到本地
        C->>C: 7  保存token<br/>切换到大厅界面

    else 登录失败
        Note over S: 密码错误或用户不存在
        S-->>C: 4  返回登录失败响应<br/>LoginResponse (fail, error_msg)
    end
```

**关键步骤解析**：
- **步骤1-3**：客户端发送登录请求，服务器从MySQL查询用户信息
- **步骤4**：验证密码
- **步骤5-6**：登录成功后将用户状态缓存到Redis，支持快速查询在线用户
- **步骤7**：返回Token供后续请求验证身份

### 3.2 创建房间流程

**流程说明**：玩家在大厅创建房间，服务器生成唯一房间号并存储到Redis，然后广播更新房间列表给所有在线用户。

```mermaid
sequenceDiagram
    participant C as 客户端<br/>(房主)
    participant S as 服务器<br/>(RoomManager)
    participant R as Redis缓存<br/>(房间数据)

    Note over C: 用户点击"创建房间"
    C->>S: 1. 发送创建房间请求<br/>CreateRoomRequest (room_name, password)

    Note over S: 生成6位随机房间号
    S->>S: 2. 生成唯一房间号<br/>room_id = random(100000-999999)

    Note over S,R: 存储房间信息到Redis
    S->>R: 3. 存储房间详细信息<br/>HSET room:{room_id} (name, owner, password...)
    S->>R: 4. 添加到活跃房间集合<br/>SADD rooms:active {room_id}
    S->>R: 5. 设置房主信息<br/>SET room:owner:{room_id} {user_id}

    Note over S: 创建成功，返回房间信息
    S-->>C: 6. 返回创建成功响应<br/>CreateRoomResponse (success, room_id, room_info)

    Note over C: 客户端切换界面
    C->>C: 7. 切换到房间界面<br/>显示房间号和等待玩家

    Note over S: 通知所有大厅用户
    S->>All: 8. 广播房间列表更新<br/>BroadcastRoomList (new_room_info)
```

**关键步骤解析**：
- **步骤2**：使用随机算法生成6位数字房间号，确保唯一性
- **步骤3-5**：将房间信息存储到Redis，包括房间名、房主、密码（可选）等
- **步骤8**：使用广播机制通知所有在大厅的用户，实时更新房间列表

### 3.3 加入房间流程

**流程说明**：玩家通过房间号加入房间，服务器验证房间状态、密码等条件，成功后将玩家添加到房间并通知房间内所有成员。

```mermaid
sequenceDiagram
    participant C as 客户端<br/>(玩家2)
    participant S as 服务器<br/>(RoomManager)
    participant R as Redis缓存<br/>(房间数据)

    Note over C: 用户输入房间号和密码
    C->>S: 1. 发送加入房间请求<br/>JoinRoomRequest (room_id, password)

    Note over S: 从Redis查询房间信息
    S->>R: 2. 获取房间数据<br/>HGET room:{room_id}

    alt 房间不存在
        Note over S: 房间ID无效
        S-->>C: 3a. 返回失败响应<br/>JoinRoomResponse (fail, "房间不存在")

    else 房间已满（2人）
        Note over S: player1和player2都已有人
        S-->>C: 3b. 返回失败响应<br/>JoinRoomResponse (fail, "房间已满")

    else 密码错误
        Note over S: 加密房间密码验证失败
        S-->>C: 3c. 返回失败响应<br/>JoinRoomResponse (fail, "密码错误")

    else 成功加入
        Note over S,R: 更新房间成员信息
        S->>R: 4. 添加到房间成员集合<br/>SADD room:{room_id}:members {user_id}
        S->>R: 5. 设置第二个玩家<br/>HSET room:{room_id} player2 {user_id}

        Note over S: 返回加入成功
        S-->>C: 6. 返回成功响应<br/>JoinRoomResponse (success, room_info)

        Note over S: 通知房间内所有玩家
        S->>Room: 7. 广播玩家加入通知<br/>BroadcastToRoom (PlayerJoined, player2_info)

        Note over C: 切换到房间界面
        C->>C: 8. 切换到房间界面<br/>显示房主和自己的信息
    end
```

**关键步骤解析**：
- **步骤2**：先从Redis查询房间是否存在，快速响应
- **步骤3a-3c**：多种失败情况的判断：房间不存在、已满、密码错误
- **步骤4-5**：将新玩家添加到房间成员集合和玩家位置
- **步骤7**：通知房间内的房主，显示"玩家2已加入"

### 3.4 对战流程

**流程说明**：双方玩家轮流落子，服务器验证每一步的合法性并检查胜负，同时实时同步棋局状态给所有观战者。

```mermaid
sequenceDiagram
    participant P1 as 玩家1<br/>(先手)
    participant S as 服务器<br/>(GameController)
    participant P2 as 玩家2<br/>(后手)
    participant Obs as 观战者<br/>(Spectators)

    Note over P1: 房主点击"开始游戏"
    P1->>S: 1. 发送开始游戏请求<br/>StartGameRequest
    S-->>P1: 2. 返回开始响应<br/>StartGameResponse

    Note over S: 通知所有相关用户
    S->>P2: 3. 游戏开始通知<br/>GameStartedNotify (先手:P1)
    S->>Obs: 4. 游戏开始通知<br/>GameStartedNotify (观战模式)

    loop 每一步对弈（轮流进行）
        Note over P1: 玩家1思考并落子
        P1->>S: 5. 发送落子请求<br/>MoveRequest (row, col)

        Note over S: 验证落子合法性
        S->>S: 6. 验证位置是否为空<br/>检查是否在棋盘内

        Note over S: 检查游戏是否结束
        S->>S: 7. 检查五子连珠<br/>判断是否有玩家获胜

        Note over S: 返回落子结果
        S-->>P1: 8. 返回落子成功<br/>MoveResponse (success)

        Note over S: 实时同步棋局状态
        S->>P2: 9. 对手落子通知<br/>OpponentMoveNotify (row, col, next_turn:P2)
        S->>Obs: 10. 棋步广播通知<br/>MoveNotify (player:P1, row, col)

        alt 游戏结束（五子连珠）
            Note over S: 更新战绩到数据库
            S->>S: 11. 保存对局记录<br/>UPDATE users SET win_count...

            Note over S: 通知所有相关用户游戏结果
            S->>P1: 12. 游戏结束通知<br/>GameOverNotify (winner:P1, reason:五子连珠)
            S->>P2: 13. 游戏结束通知<br/>GameOverNotify (winner:P1, reason:五子连珠)
            S->>Obs: 14. 游戏结束通知<br/>GameOverNotify (winner:P1, reason:五子连珠)
        end

        Note over P2: 轮到玩家2落子
        Note right of P2: 继续循环，角色互换
    end
```

**关键步骤解析**：
- **步骤5-7**：服务器验证每一步的合法性（位置是否为空、是否在棋盘内），并实时检查胜负
- **步骤9-10**：使用广播机制同步棋局状态给对手和所有观战者
- **步骤11-14**：游戏结束时更新MySQL中的战绩，并通知所有相关用户
- **循环机制**：双方轮流落子，直到分出胜负或平局

### 3.5 随机匹配流程

**流程说明**：玩家选择随机匹配，服务器将其加入匹配队列，当队列中有两个或以上玩家时自动配对并创建对战房间。

```mermaid
sequenceDiagram
    participant C1 as 客户端1<br/>(等待匹配)
    participant C2 as 客户端2<br/>(等待匹配)
    participant S as 服务器<br/>(MatchManager)
    participant R as Redis缓存<br/>(匹配队列)

    Note over C1: 玩家1点击"随机匹配"
    C1->>S: 1. 发送匹配请求<br/>RandomMatchRequest

    Note over S,R: 加入匹配队列
    S->>R: 2. 加入队列尾部<br/>LPUSH match_queue {user1_id}

    Note over S: 队列中只有1人，继续等待
    S-->>C1: 3. 返回等待响应<br/>MatchWaitingResponse (队列位置:1)

    Note over C2: 玩家2点击"随机匹配"
    C2->>S: 4. 发送匹配请求<br/>RandomMatchRequest

    Note over S,R: 玩家2加入队列
    S->>R: 5. 加入队列尾部<br/>LPUSH match_queue {user2_id}

    Note over S,R: 检查队列，发现有2人可配对
    S->>R: 6. 获取队列前2人<br/>LRANGE match_queue 0 1
    R-->>S: 7. 返回匹配的玩家<br/>[user1_id, user2_id]

    Note over S: 创建临时对战房间
    S->>S: 8. 创建匹配房间<br/>room_id = createMatchRoom(user1, user2)

    Note over S,R: 从队列移除已匹配的用户
    S->>R: 9. 清理匹配队列<br/>LPOP match_queue (2次)

    Note over S: 通知双方匹配成功
    S->>C1: 10. 匹配成功通知<br/>MatchSuccessNotify (room_id, opponent:user2)
    S->>C2: 11. 匹配成功通知<br/>MatchSuccessNotify (room_id, opponent:user1)

    Note over C1: 自动进入对战房间
    C1->>C1: 12. 切换到房间界面<br/>显示对手信息

    Note over C2: 自动进入对战房间
    C2->>C2: 13. 切换到房间界面<br/>显示对手信息
```

**关键步骤解析**：
- **步骤2-3**：第一个玩家加入队列后进入等待状态，显示"正在匹配中..."
- **步骤6-7**：使用Redis的LRANGE命令检查队列头部，发现有2人时触发配对
- **步骤8**：自动创建临时房间，无需玩家手动创建
- **步骤9**：从队列中移除已配对的玩家，防止重复匹配
- **匹配策略**：采用先进先出（FIFO）策略，确保公平性

### 3.6 聊天消息流程

**流程说明**：支持三种聊天模式（大厅聊天、私聊、房间聊天），消息通过Redis缓存并实时广播给目标用户。

```mermaid
sequenceDiagram
    participant C as 发送者<br/>(Client)
    participant S as 服务器<br/>(ChatManager)
    participant R as Redis缓存<br/>(消息历史)
    participant T as 接收者<br/>(Target)

    alt 大厅聊天（公共频道）
        Note over C: 用户在大厅发送消息
        C->>S: 1a. 发送大厅聊天请求<br/>ChatMessageRequest (type=LOBBY, content)

        Note over S,R: 保存到大厅聊天历史
        S->>R: 2a. 存储消息到有序集合<br/>ZADD chat:lobby {timestamp} {msg_json}

        Note over S: 广播给所有在线用户
        S->>All: 3a. 广播大厅消息<br/>BroadcastToLobby (sender, content, timestamp)

    else 私聊（一对一）
        Note over C: 用户选择目标用户私聊
        C->>S: 1b. 发送私聊请求<br/>ChatMessageRequest (type=PRIVATE, target_id, content)

        Note over S,R: 双向存储聊天记录
        S->>R: 2b. 存储到双方聊天历史<br/>ZADD chat:private:{sender}:{target} {timestamp} {msg}

        Note over S: 发送给目标用户
        S->>T: 3b. 私聊消息通知<br/>PrivateChatNotify (sender_info, content)

        Note over S: 确认发送成功
        S-->>C: 4b. 返回发送成功<br/>ChatMessageResponse (success)

    else 房间聊天（房间内）
        Note over C: 用户在房间内发送消息
        C->>S: 1c. 发送房间聊天请求<br/>ChatMessageRequest (type=ROOM, room_id, content)

        Note over S,R: 保存到房间聊天历史
        S->>R: 2c. 存储消息到房间记录<br/>ZADD chat:room:{room_id} {timestamp} {msg_json}

        Note over S: 广播给房间内所有成员
        S->>Room: 3c. 广播房间消息<br/>BroadcastToRoom (sender, content, timestamp)
    end
```

**关键步骤解析**：
- **大厅聊天**：使用广播机制，所有在大厅的用户都能收到消息
- **私聊**：点对点通信，使用双向存储确保双方都能查看历史记录
- **房间聊天**：只有房间内成员和观战者能看到消息
- **消息持久化**：使用Redis的有序集合（ZADD）按时间戳存储，支持历史消息查询
- **消息格式**：包含发送者信息、内容、时间戳等，便于客户端渲染

### 3.7 观战流程

**流程说明**：用户可以观看正在进行的对局，服务器发送历史棋步并实时同步后续的每一步落子。

```mermaid
sequenceDiagram
    participant Obs as 观战者<br/>(Spectator)
    participant S as 服务器<br/>(SpectatorManager)
    participant R as Redis缓存<br/>(房间/棋步数据)

    Note over Obs: 用户点击"观战"按钮
    Obs->>S: 1. 发送观战请求<br/>SpectateRequest (room_id)

    Note over S,R: 查询房间状态
    S->>R: 2. 获取房间信息<br/>HGET room:{room_id} state

    alt 房间游戏进行中
        Note over S,R: 添加到观战列表
        S->>R: 3a. 添加观战者<br/>SADD room:{room_id}:spectators {user_id}

        Note over S,R: 获取历史棋步
        S->>R: 4a. 查询所有历史步骤<br/>LRANGE room:{room_id}:moves 0 -1
        R-->>S: 5a. 返回历史棋步列表<br/>[{row:5,col:9}, {row:6,col:9}, ...]

        Note over S: 返回观战成功响应
        S-->>Obs: 6a. 观战成功响应<br/>SpectateResponse (success, history_moves, players_info)

        Note over Obs: 客户端恢复棋局状态
        Obs->>Obs: 7a. 根据历史棋步<br/>重建当前棋盘状态

        loop 实时同步后续棋步
            Note over S: 玩家每落一子
            S->>Obs: 8a. 实时棋步通知<br/>MoveNotify (player_id, row, col)

            Note over Obs: 更新棋盘显示
            Obs->>Obs: 9a. 更新本地棋盘<br/>显示新落下的棋子
        end

    else 房间未开始或已结束
        Note over S: 房间状态不符合观战条件
        S-->>Obs: 3b. 观战失败响应<br/>SpectateResponse (fail, "游戏未开始或已结束")
    end
```

**关键步骤解析**：
- **步骤2**：先检查房间状态，只有GAMING状态才允许观战
- **步骤3-5**：将观战者添加到Redis集合，并获取所有历史棋步
- **步骤7**：客户端根据历史棋步重建棋盘，让观战者看到当前局势
- **步骤8-9**：建立实时同步通道，每当玩家落子，观战者立即收到通知
- **观战特性**：观战者只能观看，无法操作棋盘，也无法影响对局

```mermaid
        -handleError() 处理错误
    }

    %% Acceptor - 接受器
    class Acceptor {
        <<Acceptor>>
        +接受新连接

        -acceptSocket: Socket 监听套接字
        -acceptChannel: Channel 监听Channel
        -listening: bool 是否正在监听

        +listen() 开始监听
        +handleRead() 接受新连接

        -newConnectionCallback 新连接回调函数
    }

    %% 关系定义
    EventLoop "1" *-- "1" Poller : 拥有 >
    EventLoop "1" o-- "*" Channel : 管理 >
    Poller "1" o-- "*" Channel : 监听 >
    Channel "1" *-- "1" TcpConnection : 关联 >
    Acceptor "1" *-- "1" Channel : 拥有 >
    Acceptor ..> EventLoop : 使用 >

    %% 注释说明
    note for EventLoop "事件循环主体\n1. 调用poll等待事件\n2. 处理活跃事件\n3. 执行待处理任务"
    note for Poller "封装epoll_wait\n返回就绪的Channel列表\n支持水平触发和边缘触发"
    note for Channel "封装fd和回调\n根据事件类型调用对应回调\n连接EventLoop和业务逻辑"
    note for TcpConnection "管理TCP连接生命周期\n缓冲区管理\n优雅关闭支持"
```

**Reactor模式工作流程**：
1. **EventLoop::loop()**：主循环调用 `poller->poll()` 等待事件
2. **Poller::poll()**：使用 `epoll_wait()` 监听所有注册的fd
3. **事件就绪**：`epoll_wait()` 返回活跃的Channel列表
4. **Channel::handleEvent()**：根据事件类型调用对应回调函数
5. **TcpConnection**：在回调中处理实际的读写业务逻辑

**关键优势**：
- **高并发**：一个线程通过epoll管理成千上万个连接
- **非阻塞**：所有I/O操作都是非阻塞的，不会阻塞主循环
- **事件驱动**：基于事件触发，减少无效的轮询开销
